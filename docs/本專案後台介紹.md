## 2. 一句話認識本後台

Brainwave 後端是以 **Spring Boot 3.x + Java 17** 為核心的多模組專案，目標是提供**乾淨分層、可重用基礎能力與穩定擴充方式**，讓後續業務模組能快速成型。

- **技術棧**：Spring Boot、Java 17、MySQL；建表由 Flyway 管理（V1～V3）。
- **模組**：`brainwave-core`（共用基礎設施）、`brainwave-service`（業務邏輯）、`brainwave-backend`（API 與啟動）。
- **設計重點**：統一回應與錯誤處理、認證與權限可替換（Mock／JWT）、設定集中有型別、稽核可落 log 或 DB、請求可追蹤、API 規格可產出給前端使用。

## 3. 架構總覽

請求進入後端的典型流程：

1. **CorrelationIdFilter**：為請求產生或沿用 `X-Request-ID`，寫入 MDC，方便整段請求的 log 用同一 ID 追蹤。
2. **CORS**：依 `app.cors.*` 決定是否放行跨域（設定見 設定治理）。
3. **AuthGuardInterceptor**：依路徑與設定判斷是否需要登入、後台或前台 scope／role；通過後將 principal 放入 request。
4. **Controller → Service**：業務邏輯在 service 層；Controller 透過 `BaseController` 使用統一的 `Result<T>` 回應格式，錯誤由 `GlobalExceptionHandler` 統一處理。

**完整流程圖（含 CORS／Guard 失敗、例外、稽核）與三層模組、啟動時序**：見 **docs/架構與請求流程.md**（內含多張 Mermaid 圖）。

架構上再補充兩點：

- **健康檢查**：`GET /api/health`，回傳狀態與版本，供部署與監控使用。
- **資料庫**：Flyway 依 `brainwave-backend/src/main/resources/db/migration` 的 V1～V3 建立 users、members、audit_log、system_config 等表；啟動時自動執行未跑過的 migration。

## 5. 各功能章節

### 5.1 統一回應與錯誤處理

所有成功回應都包在 `Result<T>` 裡（含 `success`、`data`、`message` 等），錯誤則由 `GlobalExceptionHandler` 轉成統一結構並帶錯誤碼與訊息，前端與呼叫端行為可預期。

**為什麼重要**：格式一致能降低前後端對接成本，也方便日後替換成企業規格。

**名詞**：**Result**＝統一成功／失敗的回應包裝；**ResultMapper**＝組裝這份格式的介面，可替換實作；**GlobalExceptionHandler**＝攔截例外並轉成統一錯誤回應。

**怎麼運作**：Controller 透過 `BaseController` 的 `success()`／`fail()`（或注入的 `ResultMapper`）組裝回應；若改用自訂格式，實作 `ResultMapper` 並註冊為 Bean 即可。預設的 `DefaultResultMapper` 會把當前請求的追蹤 ID（MDC 的 `requestId`）帶入 `Result.correlationId`。

**技術要點**：`Result`、`ResultMapper`、`DefaultResultMapper`、`BaseController`、`GlobalExceptionHandler`（core）；錯誤碼與訊息型別見 core 的 exception 套件。詳見 **docs/統一回應格式說明.md**、**README.md**。

### 5.2 認證與登入

後台與前台各有登入入口（`/api/auth/login`、`/api/front/auth/login`），登入後取得 token，後續 API 以 Bearer token 帶入；驗證邏輯可替換，預設為 Mock（開發用），正式環境可改為 JWT。

**為什麼重要**：開發時不需接真實 IdP 即可跑流程；上線時改設定與 profile 即可切到 JWT，不綁死實作。

**名詞**：**AuthFacade**＝登入與 token 驗證的入口；**TokenVerifier**＝解析並驗證 token、產出身分與角色；**TokenPrincipal**＝(scope, subject, role)，供權限判斷使用。

**怎麼運作**：登入由 AuthFacade 委派 UserService／MemberService 與 TokenVerifier；之後每個受保護請求由 `AuthGuardInterceptor` 呼叫 `AuthFacade.verifyToken()`，再依 TokenPrincipal 的 scope／role 決定是否放行。Mock 規則如 `demo-`*→後台、`front-demo-*`→前台；JWT 時改為 `spring.profiles.active=jwt` 並設定 `app.auth.jwt.secret`，使用專案內建的 `JwtTokenVerifier`。

**技術要點**：AuthFacade、TokenVerifier、MockTokenVerifier、JwtTokenVerifier（backend）；TokenPrincipal（core）。詳見 **docs/認證與Token說明.md**。

### 5.3 權限守衛與 RBAC

哪些路徑需要登入、需要後台或前台、要不要排除登入入口，都由設定決定；新增 API 時只要改設定，不必改 Java。

**為什麼重要**：避免未授權存取，且擴充新 API 時不用到處改程式。

**名詞**：**AuthGuardInterceptor**＝在請求進入 Controller 前檢查 token 與 path；**RBAC**＝依角色（ADMIN、EDITOR、VIEWER、MEMBER）決定可否存取；**scope**＝後台(ADMIN_*) 或前台(FRONT_*)。

**怎麼運作**：Guard 讀取 `app.auth.guard.`*（如 `admin-path-prefixes`、`front-path-prefixes`、`front-exclude-prefixes`），請求路徑符合哪一類就要求對應的 scope／role；通過後將 TokenPrincipal 放入 request，Controller 可取得當前使用者。前端可依 `principal.role` 控制選單顯示。

**技術要點**：AuthGuardInterceptor、AuthProperties.guard（core）；路徑預設含 `/api/users`、`/api/members`、`/api/system-configs`、`/api/front/`，排除 `/api/front/auth`。詳見 **docs/權限守衛與RBAC說明.md**。

---

### 5.4 設定治理

所有業務設定收斂在 `app.`* 下，用型別化物件（`@ConfigurationProperties`）存取，不散落 `@Value` 字串；環境差異用 profile（如 local、prod）與環境變數注入。

**為什麼重要**：設定有規則、有型別、可驗證，部署與除錯時較好控管。

**名詞**：**Config Layer**＝設定從環境／profile 進入，程式只透過型別化類別存取；**app.***＝本專案業務設定前綴（auth、cors、storage、audit 等）。

**怎麼運作**：設定類別集中在 core 的 `config.properties`（AuthProperties、CorsProperties、StorageProperties、AuditProperties 等）；Guard、CORS、稽核等皆讀取這些類別。IDE 可透過 `META-INF/additional-spring-configuration-metadata.json` 辨識 `app.`* 鍵。

**技術要點**：AuthProperties、CorsProperties、StorageProperties、AuditProperties；命名 kebab-case；CORS 來源由 `app.cors.allowed-origins` 等控制。詳見 **docs/設定治理說明.md**。

### 5.5 稽核日誌

在需要記錄的 API 上標註 `@Audit`，系統會自動記錄「誰、何時、打了哪個 API、成功或失敗、耗時」；可只輸出到 log，或寫入資料庫供查詢與合規使用。

**為什麼重要**：事後追查與合規需要可追溯的操作紀錄。

**名詞**：**@Audit**＝標在方法上的註解；**AuditAspect**＝AOP 切面，在方法執行後寫入一筆紀錄；**sink**＝輸出目標（console 或 db）。

**怎麼運作**：啟用 `app.audit.enabled=true`，並以 `app.audit.sink` 選 `console` 或 `db`。標了 `@Audit(action="...", resource="...")` 的方法執行完後，Aspect 會組出 action、resource、method、path、success、durationMs、requestId、principal 等，交給 `AuditLogService`；console 時打 log，db 時寫入 `audit_log` 表（需 Flyway V2）。

**技術要點**：@Audit、AuditAspect、AuditLogService、ConsoleAuditLogService、DbAuditLogService（core）；audit_log 表。詳見 **docs/稽核日誌說明.md**。

---

### 5.6 檔案儲存抽象

上傳檔案時不直接寫死本機或雲端，而是透過 `StorageService` 介面；目前實作為本機目錄，之後可替換為 S3/MinIO 等而不改呼叫端。

**為什麼重要**：先在本機跑通流程，上線再切雲端，降低綁定。

**名詞**：**StorageService**＝介面（store、toPublicUrl）；**LocalStorageService**＝本機實作，依 `app.storage.provider=local` 與 `app.storage.local.base-path` 寫入檔案。

**怎麼運作**：呼叫 `store(pathUnderBase, inputStream)` 回傳相對路徑；`toPublicUrl(storedPath)` 依是否設定 `publicBaseUrl` 產出對外 URL。錯誤以 `StorageException` 拋出。

**技術要點**：StorageService（core）、LocalStorageService（service）、StorageProperties。詳見 **docs/檔案儲存說明.md**。

---

### 5.7 系統設定 API

功能開關、公告、每頁筆數等 Key-Value 設定存於資料庫，透過 REST API 查詢與更新，不需改程式即可調整。

**為什麼重要**：營運可緊急關閉功能或調整參數，無需重部署。

**名詞**：**system_config**＝Flyway V3 建立的表；**SystemConfigController**＝提供 by-key、search、POST、PUT 等端點。

**怎麼運作**：路徑 `/api/system-configs` 已納入 Guard 後台路徑，需後台登入；依 key 或 id 查詢、分頁搜尋、新增與更新皆透過同一套 API。若需對外開放部分 key（如公告），可另開白名單 API 或放行特定路徑。

**技術要點**：SystemConfigController、SystemConfigService、system_config 表（V3）。詳見 **docs/系統設定API說明.md**。

### 5.8 資料匯出

列表資料可匯出為 CSV，符合 RFC 4180、UTF-8 BOM，Excel 可正確開啟；不依賴 Excel 專用套件。

**為什麼重要**：後台常需「匯出目前列表」供離線檢視或二次處理。

**名詞**：**CsvExportHelper**＝core 提供的靜態工具（writeCsv、toCsvBytes）。

**怎麼運作**：Controller 取得列表後轉成標題列與資料列，呼叫 `CsvExportHelper.toCsvBytes(headers, rows)` 產出位元組，設定 `Content-Type`、`Content-Disposition` 後回傳。目前 UserController 提供 `GET /api/users/export` 示範（需後台權限）。若要 xlsx，可另接 Apache POI，流程相同。

**技術要點**：CsvExportHelper（core.export）；示範端點 `/api/users/export`。詳見 **docs/資料匯出說明.md**。

### 5.9 請求追蹤

每個 HTTP 請求都有唯一 ID（X-Request-ID），會寫入 log、response header 以及 API 回應 body 的 `correlationId`，方便前後端與日誌對齊排查問題。

**為什麼重要**：出錯時可用同一 ID 過濾整段請求的 log，或請使用者提供 ID 對應後端紀錄。

**名詞**：**Correlation ID / Request ID**＝同一請求的追蹤識別碼；**MDC**＝ThreadLocal 鍵值，供 log  pattern 使用（如 `[%X{requestId}]`）。

**怎麼運作**：`CorrelationIdFilter` 讀取 request header `X-Request-ID`，若無則產生 UUID；寫入 MDC 的 `requestId`、回寫 response header，並讓 `DefaultResultMapper` 把該值帶入 `Result.correlationId`。前端可選帶入自訂 ID 或記錄回應裡的 correlationId。

**技術要點**：CorrelationIdFilter（backend）；MDC、Result.correlationId。詳見 **docs/請求追蹤說明.md**。

### 5.10 API 契約與 OpenAPI

後端提供 OpenAPI 3 規格（`/v3/api-docs`）與 Swagger UI，前端可依此產生型別或 API client（如 openapi-typescript 產出 schema.d.ts）。

**為什麼重要**：規格與程式同步，減少手寫文件與型別不同步的問題。

**名詞**：**OpenAPI**＝API 描述規格；**springdoc**＝本專案用來產出規格與 Swagger UI 的函式庫。

**怎麼運作**：backend 引入 springdoc-openapi，啟動後可訪問 `/v3/api-docs`（JSON）與 Swagger UI。若前端從不同網域請求，需在 `app.cors.allowed-origins` 放行；若希望不需登入即可看文件，Guard 不要將這些路徑列入受保護前綴。

**技術要點**：springdoc、/v3/api-docs、Swagger UI；前端型別產生腳本通常在前端 repo。詳見 **docs/OpenAPI與契約說明.md**。

### 5.11 Identity 與模組模板

專案內建 User（後台帳號）與 Member（前台會員）雙軌範例；同時提供目錄骨架與命名規範，讓新專案可選擇單軌（一套帳號）或雙軌，或從現有範例整併成一套。

**為什麼重要**：示範「帳號模組」怎麼拆、怎麼接認證與權限；新專案可複製骨架或依指引移除／整併。

**名詞**：**單軌**＝只保留一套帳號模型；**雙軌**＝後台帳號與前台會員分離；**identity/template**＝service 層的模板套件（Entity/DTO/VO/Request/Repository/Converter/Service 介面範例）。

**怎麼運作**：目前登入、Guard、API 皆支援後台與前台兩套；新模組若需受保護，需在 `app.auth.guard.`* 註冊路徑。若要收斂成單一 Identity，可依身分模組文件中的步驟整併 user/member，再刪除不需的程式與表。

**技術要點**：user、member 套件；identity/template；AuthFacade、Guard 與路徑設定。詳見 **docs/身分模組模板.md**。

### 5.12 版本化與安全

目前 API 為無版本路徑（如 `/api/users`）；若未來需支援多版並存或破壞性變更，專案內有**策略說明**（路徑前綴、Header 等）與**安全與限流**指引（安全回應標頭、速率限制選項），實作可依需求再落地。

**為什麼重要**：先約定做法，需要時再實作，避免臨時決策不一致。

**名詞**：**API 版本化**＝以路徑或 Header 區分 v1/v2；**安全標頭**＝X-Content-Type-Options、X-Frame-Options、HSTS 等；**限流**＝依 IP 或使用者限制請求頻率，可回 429。

**怎麼運作**：目前僅文件約定；若引入 v1，需在 Controller 或前綴加上版本、並更新 Guard 路徑；安全標頭與限流可透過 Filter 或現成套件（如 Bucket4j）實作，Guard 與 CORS 已存在，限流建議放在 Guard 之前。

**技術要點**：見 **docs/API版本化策略.md**、**docs/安全與限流說明.md**。

### 5.13 分層與模組

程式分為三層：**core**（共用基礎設施）、**service**（業務邏輯與資料存取）、**backend**（HTTP API 與啟動）。依賴單向：backend → service → core，新功能通常落在 service 與 backend，共用邏輯放在 core。

**為什麼重要**：邊界清楚，業務邏輯可獨立測試與演進，基礎能力可多專案共用。

**名詞**：**core**＝例外、Result、BaseController、轉換器基底、設定類別、稽核介面、CsvExportHelper 等；**service**＝Entity、Repository、Service、Converter、DTO/VO/Request；**backend**＝Controller、Filter、Guard、TokenVerifier、啟動類、Flyway migrations。

**怎麼運作**：請求進 backend，經 Filter／Guard 後到 Controller；Controller 呼叫 service 層，service 使用 core 的 Result、例外、轉換器基底等。部署時以 backend 為可執行模組，core 與 service 為依賴。

**技術要點**：見 **README.md**、**專案目錄結構.md**。

### 5.14 DB 版本管理

資料表結構由 Flyway 管理，migration 檔放在 `brainwave-backend/src/main/resources/db/migration`，依 V1、V2、V3 順序執行；已跑過的不會再執行，新變更請新增 V4、V5…，不修改既有檔。

**為什麼重要**：環境可重現、變更可追溯，多人協作與部署一致。

**名詞**：**Flyway**＝依檔名版本號依序執行 SQL 的套件；**migration**＝一支對應一版 schema 變更的 SQL 檔。

**怎麼運作**：啟動時 Flyway 檢查 `flyway_schema_history`，只執行尚未套用的 migration。目前 V1 建 users、members；V2 建 audit_log；V3 建 system_config。清空 DB 後重啟會從頭跑一輪。

**技術要點**：db/migration/V1～V3；路徑與表結構見 **docs/資料庫遷移說明.md**。

### 5.15 轉換器基底

Entity、DTO、VO、Request 之間的轉換有統一介面 `BaseConverter`，各模組用 MapStruct 實作並可針對單一功能覆寫欄位或方法，避免重複手寫與不一致。

**為什麼重要**：轉換邏輯集中、可測試，新增欄位或 Request 型別時有明確契約可依。

**名詞**：**BaseConverter**＝core 定義的介面（toDto、toEntity、toVo、toEntityFromRequest 等）；**MapStruct**＝編譯期產生實作程式碼的函式庫。

**怎麼運作**：各模組定義繼承 BaseConverter 的介面並加 `@Mapper(componentModel = "spring")`，以 `@Mapping` 指定忽略或對應欄位（如 id、password）；若有「更新用」Request，可新增 `updateEntityFromUpdateRequest` 等方法，不影響基底介面。

**技術要點**：BaseConverter（core.converter）；UserConverter、MemberConverter、SystemConfigConverter 等。詳見 **docs/轉換器說明.md**。
