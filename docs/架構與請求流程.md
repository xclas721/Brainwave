# 架構與請求流程（完整版）

本文件以圖與要點說明請求從進到出的完整路徑、三層模組關係、以及啟動與設定如何銜接。細節名詞與實作見 **本專案後台介紹.md** 各節與 **docs/** 對應說明檔。

---

## 1. HTTP 請求完整流程

下圖涵蓋：追蹤 ID、CORS、Guard、Controller/Service、成功與例外回應、以及標了 `@Audit` 時的稽核寫入。

```mermaid
flowchart TB
    subgraph 進入
        A[HTTP 請求] --> B[CorrelationIdFilter]
        B --> B1[產生/沿用 X-Request-ID<br>寫入 MDC]
        B1 --> C[CORS 檢查]
    end

    subgraph CORS
        C --> C1{來源是否允許?}
        C1 -->|否| R1[403 Forbidden]
        C1 -->|是| D[AuthGuardInterceptor]
    end

    subgraph Guard
        D --> D1{路徑需驗證?}
        D1 -->|否| E[Controller]
        D1 -->|是| D2{有 Bearer Token?}
        D2 -->|否| R2[401 Unauthorized]
        D2 -->|是| D3[TokenVerifier 驗證]
        D3 --> D4{scope/role 符合?}
        D4 -->|否| R2
        D4 -->|是| D5[principal 放入 request]
        D5 --> E
    end

    subgraph 業務
        E --> E1[呼叫 Service]
        E1 --> F[Service 層]
        F --> F1[Repository / 其他]
        F1 --> F
        F --> E
    end

    subgraph 回應
        E --> E2{有例外?}
        E2 -->|是| G[GlobalExceptionHandler]
        G --> G1[轉成統一錯誤格式]
        G1 --> R3[錯誤回應]
        E2 -->|否| H[ResultMapper 組裝 Result]
        H --> H1[帶入 correlationId]
        H1 --> R4[成功回應]
    end

    subgraph 稽核［若方法有 @Audit］
        E -.->|方法執行後| I[AuditAspect]
        I --> I1[組 AuditRecord]
        I1 --> I2{app.audit.sink?}
        I2 -->|console| I3[ConsoleAuditLogService]
        I2 -->|db| I4[DbAuditLogService]
        I3 --> I5[寫 log]
        I4 --> I6[寫 audit_log 表]
    end
```

**流程摘要**：

| 階段 | 元件 | 失敗或例外時 |
|------|------|--------------|
| 追蹤 | CorrelationIdFilter | 無，必過；僅寫入 ID |
| 跨域 | CORS（CorsConfig） | 來源不在白名單 → 403 |
| 權限 | AuthGuardInterceptor | 缺 token 或驗證失敗 / role 不符 → 401 |
| 業務 | Controller → Service | 拋出例外 → GlobalExceptionHandler → 統一錯誤回應 |
| 稽核 | AuditAspect（AOP） | 僅在方法有 `@Audit` 且 `app.audit.enabled=true` 時執行，不影響回應 |

## 2. 三層模組關係

依賴方向：**backend → service → core**。backend 是可執行模組；service 與 core 為程式庫。

```mermaid
flowchart TB
    subgraph brainwave-backend["brainwave-backend（API 與啟動）"]
        B1[Controller]
        B2[Filter / Interceptor]
        B3[TokenVerifier 實作]
        B4[DefaultResultMapper]
        B5[BrainwaveBackendApplication]
        B6[Flyway migrations]
    end

    subgraph brainwave-service["brainwave-service（業務邏輯）"]
        S1[Service]
        S2[Repository]
        S3[Entity / DTO / VO]
        S4[Converter]
        S5[LocalStorageService 等]
    end

    subgraph brainwave-core["brainwave-core（共用基礎設施）"]
        C1[Result / ResultMapper 介面]
        C2[BaseController / PageResponse]
        C3[例外 / GlobalExceptionHandler]
        C4[BaseConverter / 設定類別]
        C5[Audit 介面 / CsvExportHelper]
        C6[StorageService 介面]
    end

    B1 --> S1
    B1 --> C2
    B2 --> C4
    B3 --> C4
    B4 --> C1
    S1 --> S2
    S1 --> S4
    S4 --> C4
    S1 --> C3
    S5 --> C6
    B5 --> B1
    B5 --> B2
    B5 --> B6
```

**各層職責簡表**：

| 模組 | 職責 | 典型內容 |
|------|------|----------|
| **backend** | 接收 HTTP、組裝回應、註冊 Filter/Guard、啟動應用 | Controller、AuthGuardInterceptor、CorrelationIdFilter、JwtTokenVerifier、WebMvcConfig、db/migration |
| **service** | 業務規則、資料存取、轉換 | UserService、SystemConfigService、Entity、Repository、Converter、LocalStorageService |
| **core** | 共用契約與工具，不依賴 service/backend | Result、ResultMapper、BaseController、GlobalExceptionHandler、AuthProperties、AuditAspect、CsvExportHelper、StorageService 介面 |

## 3. 啟動時發生什麼（含 Flyway）

```mermaid
sequenceDiagram
    participant App as BrainwaveBackendApplication
    participant Flyway as Flyway
    participant DB as 資料庫
    participant Env as 環境變數 / application-*.properties

    App->>Env: 讀取 profile（local / prod）
    Env-->>App: 設定（DB URL、app.auth.*、app.cors.* 等）
    App->>Flyway: 執行 migration
    Flyway->>DB: 查 flyway_schema_history
    DB-->>Flyway: 已套用版本
    Flyway->>DB: 執行未套用的 V1、V2、V3…
    DB-->>Flyway: 完成
    Flyway-->>App: 就緒
    App->>App: 註冊 Filter、Interceptor、Controller
    App->>App: 啟動內嵌 Tomcat，監聽 8080
```

- **設定來源**：環境變數（部署時）→ `application-{profile}.properties` → 程式透過 `@ConfigurationProperties` 型別化物件存取（如 AuthProperties、CorsProperties）。
- **Flyway**：啟動時依 `db/migration/` 內檔名順序執行尚未套用的 migration，建立或更新表；不會重跑已記錄的版本。

## 4. 稽核與請求的關係（時序）

標了 `@Audit` 的方法在**正常執行完或拋出例外後**都會由 AOP 寫入一筆稽核；不影響回傳值與例外傳遞。

```mermaid
sequenceDiagram
    participant Guard as AuthGuardInterceptor
    participant Ctrl as Controller
    participant Svc as Service
    participant Aspect as AuditAspect
    participant Sink as AuditLogService

    Guard->>Ctrl: 放行（principal 已放入 request）
    Ctrl->>Svc: 呼叫業務方法
    Svc-->>Ctrl: 回傳（或拋出例外）
    Ctrl-->>Guard: 回傳 / 拋出
    Note over Ctrl: 方法結束後（finally）
    Ctrl->>Aspect: 攔截 @Audit 方法
    Aspect->>Aspect: 組 AuditRecord（action, resource, requestId, principal…）
    Aspect->>Sink: record(record)
    Sink->>Sink: console → log；db → 寫 audit_log
```
